# VI System Code Style Rule

## 1. 适用范围
- 本规则适用于 `VI-system` 仓库的全部代码与文档。
- 重点覆盖：主题系统、变量映射、主题抽屉、Element Plus 覆盖、OpenSpec 产物。

## 2. 变量分层规则（强制）
1. 组件层只能消费语义变量层。
2. 语义变量默认前缀为 `--vi-*`，但前缀可配置，不可在组件逻辑中写死品牌前缀。
3. `--el-*` 与 `--wb-*` 仅作为映射层，不作为语义源。

## 3. 样式与 Token 规则（强制）
1. 禁止硬编码主题颜色（hex/rgb）到组件样式，统一引用变量。
2. 零魔法数：间距、圆角、阴影、字号必须来自 Token 或变量。
3. Less 色值计算优先编译期完成，运行时仅做变量切换。

## 4. 主题能力规则
1. 主题切换统一通过能力层（如 `useViTheme`）执行。
2. 暗黑模式遵循双层机制：CSS fallback + JS 注入覆盖。
3. 主题切换后必须保证 Element Plus 与 Workbench 联动。
4. V1 阶段忽略水印功能，不新增水印相关能力。

## 5. Vue / TypeScript 规则
1. Vue 组件使用 `script setup`。
2. 公共 API 与核心状态必须显式类型声明。
3. 主题能力命名使用 `useVi*` 前缀。
4. 主题注入逻辑应放在 composable，不应散落在业务组件中。

## 6. Less / CSS 规则
1. 文件职责分离：token、语义变量、映射变量、覆盖样式分文件管理。
2. Workbench 风格样式优先使用 BEM 命名。
3. `!important` 仅用于第三方覆盖冲突，不得滥用。
4. 禁止将业务页面临时 hack 样式放入公共库。

## 7. OpenSpec 规则
1. proposal/design/tasks/spec 必须使用简体中文。
2. 仅在 API/包名/变量名等技术名词处使用英文。
3. 任务拆分需可执行、可验收，与 `WORKFLOW.md` 保持一致。

## 8. 提交前检查
1. 是否有硬编码颜色残留？
2. 是否符合“语义层 -> 映射层 -> 组件消费”？
3. 新增主题是否无需改组件文件？
4. 暗黑模式切换是否无闪烁、无变量不同步？
5. OpenSpec 文档是否为中文？
