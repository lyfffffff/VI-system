## Context

当前 VI 主题系统已经完成 V1 能力（17 主题、暗黑模式、主题抽屉、Storybook 文档入口），但在 V2 进入组件级样式对齐前，存在三个结构性问题：
1. 语义变量数量偏少，导致组件改造容易直接写覆盖样式。
2. 覆盖规则集中在单文件，影响面大，回归定位困难。
3. 前缀可配置能力在 TS 层与 Less 层存在断点，可能出现“配置可改但样式不跟随”。

外部参照：`askdata-dashboard.pages.dev/data-cockpit` 页面显示其 UI 为 “Element Plus 组件骨架 + workbench 业务样式定制”，因此本次设计需同时支持通用组件与业务壳层。

约束条件：
- 技术栈保持 `Vue 3 + TypeScript + Less + Element Plus`。
- 暗黑模式机制保持 `html.dark` 与运行时变量注入协同。
- 继续排除水印功能。

## Goals / Non-Goals

**Goals:**
- 建立稳定的三层样式改造优先级：`CSS var` > `业务 class` > `直接覆盖组件内部样式`。
- 扩展语义变量层并补全到 Element Plus 映射层，使首批组件可主要依赖变量驱动。
- 将覆盖文件按组件职责拆分，降低修改冲突与回归成本。
- 明确局部主题覆盖的优先级和作用域边界（支持 `3B`：局部覆盖允许）。
- 打通前缀可配置链路，默认 `--vi-*` 但不绑定最终前缀。

**Non-Goals:**
- 不新增主题数量与主题编辑器。
- 不变更业务页面布局或业务数据逻辑。
- 不在本次变更引入服务端主题同步。

## 架构方案

主题系统分为 4 层：
1. `tokens` 层：基础设计 token（基础色、圆角、阴影、间距）。
2. `semantic-vars` 层：语义变量（surface/text/border/fill/interactive）。
3. `mapping` 层：将语义变量映射到 `--el-*` 与 `--wb-*`。
4. `overrides` 层：组件级兜底覆盖，仅处理变量不足以表达的视觉差异。

目录策略：
- 保持 `packages/vi/src/styles` 作为统一入口。
- 将 overrides 按组件拆分为子文件，由总入口按顺序引入。

## 数据 / 状态流

1. 启动：`initViTheme()` 调用 `useViTheme()` 读取本地存储并恢复 `themeKey/isDark`。
2. 注入：`useViTheme` 计算主题变量并写入 root，切换 `html.dark`。
3. 渲染：组件优先消费语义变量与映射变量。
4. 交互：`ThemeDrawer` 触发 `setTheme/setDark`，状态更新后统一回写存储并刷新变量。
5. 局部覆盖：业务在容器作用域内覆写语义变量；未覆写部分回退全局。

## 扩展性设计

- 新增主题：仅改 `theme-config`（单一数据源）及颜色推导，不改组件。
- 更换主题色：优先改 preset 或语义变量，不直接改组件样式。
- 前缀调整：通过 `useViTheme` 的 prefix 配置下发，映射与覆盖层统一消费派生变量。
- 组件扩展：新增组件改造时先补映射变量，再补最小覆盖文件。

## Decisions

### 1. 组件改造采用“变量优先，覆盖兜底”
- 决策：默认先改 `semantic-vars` 与 `element-plus-mapping`，只有变量不足才进入 overrides。
- 理由：最小破坏、升级兼容性更好。
- 备选：直接覆盖 `.el-*`。
  - 未选原因：版本升级风险高、排查困难。

### 2. 允许业务局部覆盖（3B）
- 决策：在业务容器作用域允许覆盖语义变量，优先级高于全局。
- 理由：支持业务场景差异，不破坏库级默认主题。
- 备选：只允许全局主题。
  - 未选原因：灵活性不足，业务需要二级主题能力。

### 3. 前缀保持可配置，不绑定 `--vi-`
- 决策：默认前缀仍为 `vi`，但实现需支持替换前缀并完整生效。
- 理由：符合你对语义层可迁移的要求。
- 备选：硬编码 `--vi-*`。
  - 未选原因：后续品牌或系统迁移成本高。

### 4. 覆盖样式按组件拆分
- 决策：`workbench-overrides.less` 作为聚合入口，实际规则拆分到组件文件。
- 理由：降低文件耦合，便于任务并行与回归。
- 备选：维持单文件。
  - 未选原因：后续维护成本持续升高。

## Risks / Trade-offs

- [风险] 变量映射扩展后引发旧样式回归  
  -> [缓解] 增加 Storybook 回归场景矩阵（亮/暗 + 17 色 + 首批组件）。

- [风险] 局部覆盖策略被滥用，出现“页面各自为政”  
  -> [缓解] 规定仅允许覆盖语义层变量，禁止跨层直接改 `--el-*` 作为常态方案。

- [风险] 前缀可配置链路不完整导致部分样式失效  
  -> [缓解] 加入 prefix 验收项（默认前缀与自定义前缀各验证一轮）。

- [权衡] 拆分覆盖文件会增加文件数量  
  -> [收益] 可维护性、定位效率和并行开发能力显著提升。

## Migration Plan

1. 先完成规范与任务拆解（当前 change）。
2. 实现阶段先做主题配置与类型收敛，再做变量层扩展。
3. 完成映射层后，再按组件拆分 overrides 并逐个回归。
4. 最后补齐文档与 Storybook 场景，完成验收。

回滚策略：
- 如 V2 出现大面积样式回归，回滚到归档版本 `2026-03-27-adopt-storybook-docs` 对应主题实现。
- 通过保留原入口文件与可逆 import 顺序，快速恢复旧行为。

## Open Questions

1. 首批组件之外（如 `el-menu`、`el-popover`）是否纳入同一批次？
2. 局部覆盖是否需要额外提供 `helper class`（如 `.vi-theme-scope`）降低业务接入成本？
3. 是否在 CI 增加主题快照比对，作为回归门禁？

## 修改范围治理（Style Contract）

为避免“组件越改越散”，V2 追加库级治理约束：
- 统一以 `VI_STYLE_CONTRACT.md` 作为样式修改准入标准。
- 所有样式变更必须标注改造层级：`变量层 / 组件层 / 业务层`。
- 禁止无作用域全局 `.el-*` 覆盖；局部覆盖仅允许语义变量层。
- Storybook 回归场景作为交付门槛之一，而非可选项。

该策略确保“原型级还原能力”与“低风险可维护性”同时成立。

## 高保真原型规范化（V2.1 扩展）

为支持“原型级页面还原”且避免修改范围失控，增加一层执行清单：
- 以 `docs/guides/prototype-normalization-checklist.md` 作为组件还原推进基线。
- 分为 Foundation/Layout/筛选条/指标卡/表格/图表/导航/弹层/暗黑/回归矩阵 10 个模块。
- 每个模块都要求可回归场景 + 作用域约束 + 层级归属标注。

该扩展不改变既有 `token -> semantic -> mapping -> overrides` 架构，仅增强执行与验收可操作性。
