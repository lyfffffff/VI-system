## Context

当前主题抽屉能力在 `ad-center-web-source` 中以项目内实现存在，核心能力可用，但存在以下问题：
- 主题能力与业务项目耦合，难以在其他项目复用。
- 主题变量以 `--el-*`/`--wb-*` 直接注入为主，缺少语义变量中间层。
- 组件对主题能力的接入路径不统一，缺少库级 API 契约。
- OpenSpec 层面尚未形成可验证的规范与场景定义。

约束条件：
- 继续沿用 `Vue 3 + TypeScript + Vite + Less + Element Plus`。
- V1 固定 17 预设主题。
- V1 明确忽略水印功能。
- 遵循项目代码风格与变量分层规则（`CODE_STYLE_RULE.md`）。

## Goals / Non-Goals

**Goals:**
- 将 ad-center 的主题能力迁移为 VI 库可复用模块。
- 建立“语义变量层（默认 `--vi-*`）-> `--el-*`/`--wb-*` 映射层”的标准架构。
- 提供统一对外接口：`useViTheme` 与 `<ViThemeDrawer />`。
- 保证主题状态可持久化、可恢复，切换无明显闪烁。
- 提供清晰的接入文档与迁移约束。

**Non-Goals:**
- 不实现水印相关功能。
- 不实现服务端主题同步。
- 不实现用户自定义主题编辑能力。
- 不改造业务页面结构，仅提供库能力与接入方式。

## Decisions

### 1. 采用“二次开发迁移”策略
- 决策：以 `ad-center-web-source` 现有主题实现为基础，按库分层重组。
- 理由：可复用现有成熟逻辑，降低重写成本并缩短交付周期。
- 备选方案：从零重写主题系统。
  - 未选原因：周期长、风险高、验证成本高。

### 2. 主题变量改为语义分层
- 决策：统一由语义变量层驱动，默认前缀 `--vi-*`，并保持前缀可配置。
- 理由：解耦业务组件与底层实现，便于未来前缀调整和多项目复用。
- 备选方案：继续直接以 `--el-*` 作为语义源。
  - 未选原因：与组件库耦合过强，扩展性弱。

### 3. 主题状态由能力层集中管理
- 决策：使用 `useViTheme` 统一处理主题状态、持久化、变量注入、暗黑模式切换。
- 理由：减少页面层重复逻辑，保证行为一致。
- 备选方案：组件内各自处理主题注入。
  - 未选原因：状态分散，维护成本高且容易不一致。

### 4. 抽屉组件与能力层解耦
- 决策：`<ViThemeDrawer />` 仅负责 UI 与交互，核心状态依赖 `useViTheme`。
- 理由：便于将来替换 UI（设置页、弹窗、顶部面板）而不改主题引擎。
- 备选方案：抽屉组件内部全量管理状态。
  - 未选原因：复用性差，不利于扩展。

### 5. 暗黑模式采用双层机制
- 决策：保留 CSS fallback（`html.dark`）+ JS 运行时覆盖。
- 理由：降低首屏闪烁风险，同时保持动态切换能力。
- 备选方案：仅 JS 注入。
  - 未选原因：首屏阶段存在 FOUC 风险。

### 6. 数据与状态流
- 启动阶段：`initViTheme()` 读取本地存储 -> 恢复 `themeKey`/`isDark` -> 注入语义变量与映射变量。
- 交互阶段：`ViThemeDrawer` 触发主题或模式变更 -> `useViTheme` 更新状态并持久化 -> 重新注入变量。
- 渲染阶段：组件消费语义变量与映射变量，Element Plus 与 Workbench 样式同步更新。

### 7. 扩展性策略
- 新增主题：只改 `theme-config` 与算法配置，不改组件结构。
- 改主题色：只改 preset 或 override，不改页面组件代码。
- 前缀调整：通过语义层配置能力实现，避免组件级改造。

## Risks / Trade-offs

- [风险] 迁移后变量命名不一致导致样式回归  
  -> [缓解] 建立变量映射清单与回归样例页面（按钮、输入、表格、导航）。

- [风险] 暗黑模式 fallback 与 JS 注入冲突  
  -> [缓解] 约定注入优先级与变量覆盖顺序，并增加切换回归测试。

- [风险] 从 ad-center 迁移时带入业务私有样式 hack  
  -> [缓解] 仅迁移可复用规则，业务结构样式改为文档示例，不入库。

- [风险] 组件层仍直接依赖 `--el-*` 变量  
  -> [缓解] 代码评审强制检查语义层消费规则，新增 lint/审查清单项。

## Migration Plan

1. 先迁移主题配置、类型、工具函数、能力层（M1/M2）。
2. 再迁移并重构主题抽屉组件（M3），移除水印逻辑。
3. 拆分 `theme.less` 为 token/语义/映射/覆盖文件（M4）。
4. 完成接入文档与 OpenSpec 规格，进行样式与交互回归（M5）。
5. 发布私有包并在目标项目进行灰度接入验证（M6）。

回滚策略：
- 若迁移版本出现严重样式回归，回滚到上一版 VI 包并恢复旧接入入口。
- 保留 ad-center 原实现作为应急兜底，不在同一版本同时替换全部入口。
